# Model Validation & Reference Strategy (modelRef) 採用の設計決定

このドキュメントでは、Board機能において3Dモデルなどのアセットを「実体のコピー保持」ではなく「参照（modelRef）」によって紐付ける方式を採用した理由を整理します。

## 概要

Board（My Boards, Team Boards等）に3Dモデルやレイアウトを追加する際、データのクローンやディープコピーを作成して保存するのではなく、常に元データ（正本）のIDの参照先のみを保持する設計を採用しています。

## 採用の理由

### 1. 更新におけるデータ整合性の確保 (Single Source of Truth)
元データ（`users/{uid}/models/{modelId}`）が作成者によって更新された際（例: タイトル変更、カテゴリ変更、価格変更）、コピーを持たせていると全Boardのコピーデータとの間で同期処理（Fan-out）が必要になります。参照方式であれば、表示時に常に最新のデータを引いてくるため、データの不整合が原理的に発生しません。

### 2. データ重複の回避とストレージ容量の削減
3Dモデルやその周辺メタデータを各Boardにコピーすると、Firestoreのドキュメントサイズが不必要に膨れ上がります。重い情報を一箇所（正本）にまとめることで、DB容量やそれに伴うコスト増を抑えられます。

### 3. 保守性とスケーラビリティ
Boardの中に並ぶアイテムは「ショートカットの集合」のようなものです。将来的に`model`以外にも`layout`や`present`などを統合的なBoardで扱う際、すべて参照IDで扱うことでポリモーフィックな拡張（refId, targetId, refType）が容易になります。

### 4. 権限管理の簡略化
「そのモデルを編集できる権限があるか？」は常に正本側のセキュリティルールで決定されます。Board上にコピーがあると、コピーに対する書き込み権限と正本への権限が乖離するリスクがありますが、参照方式であれば常に正本（authorId）が基準となります。

## 設計上のトレードオフと対策

**トレードオフ:**
Boardを開いた際、参照IDをもとに実体ドキュメントを並列で取得（Read操作）するため、Firestoreのリード回数やクライアント側のローディング負荷が増加する懸念があります。

**対策の方針:**
- UI側でのキャッシュ（SWR / React Query等）の活用
- FirestoreのバッチReadや、取得結果のブラウザセッションでの再利用
- 表示上最低限必要な情報（名前やサムネイルURL）のみを非正規化して参照側に持たせるキャッシュパターンの段階的導入（更新頻度が極めて低いデータに限る）
