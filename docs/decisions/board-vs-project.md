# Project と Board の違いについての設計決定

このドキュメントでは、SEKKEIYAアプリケーションにおける「Project」と「Board」の概念の違いと、なぜこれらを別々の階層として分離しているのかについて整理します。今後の機能追加や設計判断における基準として扱います。

## 概要

SEKKEIYAでは、**「Project = 案件や目的の単位」**、**「Board = 作業領域や専門領域ごとの単位」**として明確に分離しています。

## なぜ Board = Project ではないのか？

もし1つの案件に対して1つの「ボード」しか存在しないと仮定すると、モデリングの参考画像、レイアウト図、プレゼン資料、議事録のメモなどの全く異なる性質のアセットが一箇所に混在してしまいます。3Dの性質上、用途ごと（3DSS, 3DSL, 3DSPなど）に扱うデータ構造やUIが全く異なるため、それらを混在させるとUXが破綻します。

そのため、Project を最上位のコンテナとし、その中に複数の Board を束ねるアーキテクチャを採用しました。

## Project の役割
- **案件・目的のコンテナ:** 「◯◯邸リノベーション」「カフェ内装コンペ」など、チームが取り組む共通の目的を定義します。
- **権限・メンバー管理:** 誰がその案件に参加しているかを一元管理します。
- **グローバルな情報:** クライアント情報やスケジュール、大域的なメモを含みます。

## Board の役割
- **作業領域のコンテナ:** 「リビングのレイアウト検討」「客席用の参考モデル一覧」など、特定のタスクに集中するためのワークスペースです。
- **専門領域の分類:** 子アプリ（3DSL, 3DSS, 3DSP等）のデータと密結合する単位です。あるBoardを開いた時は「レイアウト作業」、別のBoardを開いた時は「プレゼン作成」といったコンテキストスイッチを提供します。
- **整理の単位:** ファイルシステムの「フォルダ」のように機能し、アセットを論理的にグループ化して再利用しやすくします。

## 設計への影響

この分離により、AI Chat や AI Drive は、「ユーザーが今どのProjectに属し、その中のどのBoardで作業しているか」を読み取ることで**重層的なコンテキスト**を獲得します。「このプロジェクトの予算で、今のこのボードのイメージに合うもの」といった高度な推論が可能になります。
