# AI記者（SEKKEIYA Reporter）設計書

**コンセプト:** SEKKEIYA の中に "記者" を常駐させ、ユーザーの制作活動から
**唯一無二の一次情報コンテンツ**を生み出し、SEOフライホイールを回す。

- 競合がコピー不可（自社ユーザーの実体験）＝ Google の E-E-A-T「Experience」を直撃
- ユーザー数に比例してコンテンツが増える＝放っておくほど強くなる
- SEO資産であると同時に「導入事例・お客様の声」マーケ資産にもなる

リスク設計のため **2モード**に分離する。①から稼働、②を本命として育てる。

---

## 全体像

```
 [製品の利用データ]                          [officialArticles CMS]
        │                                          ▲
        ├── モード①: 集計トレンド（全自動・匿名）──┤  status:draft → 人/本人が承認 → published
        │                                          │            │
        └── モード②: opt-in取材（要同意・本命）────┘            ▼
                                            既存パイプラインに乗る:
                                   generate-sitemap → prerender → indexnow
```

両モードとも出力先は既存の `officialArticles`（Firestore）。
公開すれば **動的sitemap・プリレンダ・IndexNow がそのまま拾う**ので、配信側の新規開発は不要。

---

## モード① 集計トレンド記事（全自動・🟢）

**個人を特定しない集計データだけ**から記事を生成する。プライバシーリスクが低く、無人運用できる。

### データソース（すべて匿名・集計）
- AIチャットで多かった**指示の傾向**（例: 「カフェ」「20畳」「北欧」等の頻出語）
- 生成された**レイアウトの傾向**（部屋タイプ、家具カテゴリの分布）
- 人気の**素材/モデル/テンプレ**ランキング
- 期間比較（今週 vs 先週の伸び）

### 記事の型（テンプレ）
- 「今週SEKKEIYAで作られたカフェ空間、傾向トップ3」
- 「ユーザーがAIに最も頼んだインテリア指示 TOP10」
- 「いま伸びている素材カテゴリ」
- → いずれも**長尾キーワード**を自然に含み、定期更新で鮮度シグナルも出せる

### プライバシー設計（必須）
- 出力は**集計値のみ**。個人・プロジェクト・プロンプト原文をそのまま載せない。
- **k-匿名性しきい値**: 母数が N 件（例: 20）未満のセグメントは記事化しない（特定防止）。
- 集計は事前に匿名化された分析テーブルから読む（生ログを記者に渡さない）。

### 実装スケッチ
1. 集計バッチ（Cloud Functions/Scheduler, 週次）→ `analytics/weeklyTrends` に匿名集計を書く
2. `reporter-mode1.mjs`（または Function）: 集計を読み、LLMでテンプレ記事を生成 → `officialArticles` に `status:"review"` で投入
3. 運用者がざっと確認して `published`（完全自動公開も技術的には可能だが、初期は1段レビュー推奨）

## モード② opt-in 取材記事（要同意・🟡・本命）

**"監視" ではなく "取材依頼"。** ユーザー本人の同意のもと、AI記者がインタビューし、
**本人が自分の記事を承認してから**公開する。同意・本人確認・品質レビューが導線に内包される。

### トリガー（取材を打診する好機）
- プロジェクト完成 / サイト公開 / 初めての成果物 などの達成イベント
- 「あなたの制作ストーリーを記事にしませんか？（任意）」を控えめに提示（押し売りしない）

### フロー
```
達成イベント
   ▼
取材オファー（任意・いつでも辞退可）──[辞退]──▶ 終了（記録して再勧誘を抑制）
   │[同意 = 明示的opt-in]
   ▼
AI記者インタビュー（チャット数問）
   例: 何を作った? / どんな課題を解決? / AIのどこが効いた? / 工夫した点は?
   ▼
LLMが下書き生成（事例記事 + 引用 + できれば成果物画像）
   ▼
本人レビュー画面: 編集・公開可否・氏名/会社名の表示可否・画像の使用可否 を本人が選択
   │[本人が承認]
   ▼
officialArticles に published → sitemap/prerender/indexnow が拾う
```

### 既存資産の再利用（新規開発は最小）
- **インタビュー UI** = 既存の**ツアーエンジン**（チャット自動オープン・自動入力の仕組み）の応用
- **下書き生成** = 既存の **AIスイート（LLM基盤）**
- **公開** = 既存の **officialArticles CMS + 動的sitemap + プリレンダ + IndexNow**
- → 新規で要るのは「取材オファーUI」「インタビュー進行スクリプト」「本人レビュー/同意UI」程度

### データモデル（案）
```
interviews/{id}
  uid, status: offered|accepted|declined|drafted|approved|published
  consent: { publish:bool, showName:bool, showCompany:bool, useImages:bool, at:Timestamp }
  qa: [{ q, a }]
  draftArticleId  // officialArticles への参照
officialArticles/{id}
  ...既存... + source: "reporter-interview", reviewedBy: uid（本人承認）
```

### プライバシー / 法務ガード（必須）
- **明示的 opt-in 必須**。黙ってユーザーの操作・成果物を記事化しない。
- 氏名・会社名・画像は**項目ごとに本人が個別同意**（既定はオフ）。
- **公開後の取り下げ**手段を用意（本人がいつでも非公開化できる）。
- 個人情報の取得・利用目的をプライバシーポリシーに明記。
- 未成年・匿名ゲストは対象外、または保護者同意等の追加配慮。

## 共通基盤: topicQueue とフライホイール接続

- `gsc-report.mjs`（Phase 5a）が出す**惜しいキーワード**を `topicQueue` に投入
- モード①②の記事生成時、topicQueue のキーワードを**狙い語**としてLLMに渡す
  → 「実需のある検索語 × 実ユーザーの一次体験」という最強の組み合わせになる
- 公開 → GSCにデータ蓄積 → 次の惜しいKW抽出 … でループが閉じる
