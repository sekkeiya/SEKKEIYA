# SEKKEIYA / 3DSS Ecosystem Documentation

## はじめに (目的)

SEKKEIYA および各サブアプリ（3DSS, 3DSL, 3DSP, 3DSC, 3DSB, 3DSQ）を含むエコシステム全体の設計仕様、データ構造、処理フローを蓄積するためのディレクトリです。
プロジェクトの大規模化に伴い、設計資料がどこにあるかをすぐに把握し、迷わず新規資料を追加できるように構成を統一しています。

## ディレクトリ構成
設計ドキュメントは大きく `diagrams/` と `specs/` に分かれています。

- **`diagrams/`**: MermaidやER図、構成図など「視覚的に理解するための図解」を配置します。
- **`specs/`**: 文章ベースの詳細仕様、責務定義、データ設計等を配置します。

さらに、その配下で以下のカテゴリに分類しています：

1. **`architecture/`**: 
   - システム全体構成、プロジェクト/ボード等の基盤概念
   - 共通API、責務分割、各アプリ関係図など
2. **`data/`**: 
   - Firestoreスキーマ設計 (ER図)、単一情報源 (Source of Truth) の定義方針
   - 在庫・棚卸し・削除計画・マイグレーション情報など
3. **`flows/`**: 
   - 認証フロー、Boardコンテキスト遷移フロー、データフロー
   - AI Drive、AI Chat などの基盤処理フロー仕様など
4. **`apps/`**: 
   - 3DSS, 3DSL, 3DSP, 3DSC など、個別アプリケーション固有の役割や扱うデータ設計
5. **`setup/`** (specsのみ): 
   - ローカル開発環境の構築、エミュレータ設定手順など

## 今後の資料追加ルール

1. **新規ファイルは新構成に従う**:
   上記カテゴリに従い、最も適切な場所に配置してください。「仕様は `specs/`、図は `diagrams/`」と明確に分離し、関連する図と仕様書の間で相互リンクを張る運用を推奨します。

2. **既存ファイルは段階移行する**:
   旧来から存在するルート直下のファイルや `decisions/` ディレクトリなどは、いきなり全てを移動してリンク切れ（import含む）のリスクを抱えるのではなく、リファクタリング等のタイミングで段階的に新構成へ移行（move/rename）を実施します。
