## 🏗️ 1. 旧構成のボード残骸

Unified Schema v2 (`boards`) の導入により、以下の古い構造はレガシーとなりました。

### 1-1. `users/{uid}/myBoards`
- **読み取り元ファイル:** `r3dm-share/src/shared/api/boards/boards.js` (旧マイボード表示用), `global-panel/src/api/boards/myBoards.js` (互換コード)
- **書き込み元ファイル:** 同上
- **関連 Cloud Functions:** なし
- **現在の UI 参照有無:** `global-panel` の新UI（Dashboard等）では参照しておらず不要。移行漏れの古いコンポーネントのみが参照する可能性あり。
- **削除可能と判断した根拠:** 新 `boards` コレクション（`boardType: "myBoards"`）に一元化済みであり、100%データ移行が確認できれば構造ごと削除可能。

### 1-2. `teamBoards` (ルートのグローバルコレクション)
- **読み取り元ファイル:** `r3dm-share/src/shared/api/boards/boards.js`
- **書き込み元ファイル:** `teamBoards.js` 等の旧作成フロー
- **関連 Cloud Functions:** なし
- **現在の UI 参照有無:** 原則なし（新UIは `boards` を使用）
- **削除可能と判断した根拠:** 新 `boards` コレクション（`boardType: "teamBoards"`）に移行済みのため、移行状況を確認後に削除可能。

### 1-3. `users/{uid}/teamBoards` (リンク用コレクション)
- **読み取り元ファイル:** `boards.js` (サイドバー一覧取得等)
- **書き込み元ファイル:** 招待機能の承諾時（旧実装）
- **関連 Cloud Functions:** なし
- **現在の UI 参照有無:** `useBoards` などの新フックは `boards` で `memberIds` 配列をクエリするため不要。
- **削除可能と判断した根拠:** クエリで直接自分の参加ボードを取得できるようになったため、ユーザー配下のリンクデータは役割を終えた。

## 🔍 2. 要再評価（即削除はせず要件を精査する候補）

### 2-1. `boardsPublic`
- **読み取り元ファイル:** `sekkeiya/src/shared/api/boards/public.js`
- **書き込み元ファイル:** 上記ファイルの `upsertPublicBoard`
- **関連 Cloud Functions:** なし（アクション起点でクライアントが書き込む）
- **現在の UI 参照有無:** 公開ギャラリー、未ログイン向けのディスカバリー画面などで参照されている。
- **削除可能と判断した根拠（なぜ再評価とするか？）:**
  - **構造的重複:** `boards` 上の `where("visibility", "==", "public")` で理論上は代用可能。
  - **再評価・保留の理由:**
    1. **公開一覧画面の要件 & 検索性:** `boardsPublic` には公開専用のサマリや検索用メタデータ（タグ等）が最適化されている可能性がある。
    2. **Firestore query の単純化:** 未ログインユーザー向けのセキュリティルールを `boards` で複雑に組むより、`boardsPublic` を全公開とする方が安全・シンプル。
    3. **コスト / read 数:** アクセス過多なディスカバリー画面で重い `boards` 群をスキャンさせるより、軽量なミラーリストを読ませる方がコスト減になる想定。
    4. **ミラー維持コスト:** ただし、ボード更新時の二重書き込み処理によるバグ発生リスクとのトレードオフ。
  - **結論:** 即削除対象からは外し、公開ギャラリーの設計見直し時に「`boards` を直接読む」アーキテクチャへのシフトがペイするかをテストした上で判断する。
